# 架构说明

## 1. 分层原则

仓库按“公开契约层、运行时管理层、具体模块层、测试验证层”组织：

- 公开契约层：`include/module_context/**`，面向外部使用者。
- 运行时管理层：负责模块装载、生命周期驱动、服务注册与查询。
- 具体模块层：提供某一类业务能力的实现，例如 `rabbitmq_bus`。
- 测试验证层：覆盖配置解析、生命周期、插件集成和模块行为。

对外稳定契约只有公开契约层；其余层均可随内部实现演进调整。

## 2. 核心对象职责

### `IContext`

- 代表一个模块运行时上下文。
- 负责统一驱动 `Init / Start / Stop / Fini`。
- 负责对外暴露服务查询入口 `GetService()`。
- 对外提供 `ModuleManager()` 视图，以便调用方装载模块和读取模块配置。

### `IModuleManager`

- 负责装载模块配置或单个模块动态库。
- 负责创建模块实例并校验实例名、类型名和插件版本。
- 负责按顺序执行初始化/启动，按逆序执行停止/释放。
- 负责按实例名返回模块对象和模块配置。

### `IModule`

- 表示模块实例的最小公开抽象。
- 对外提供实例名、类型名、版本号和生命周期状态。
- 供运行时管理层统一驱动。

### `IMessageBusService`

- 表示消息总线服务接口。
- 由 `rabbitmq_bus` 模块提供。
- 通过 `IContext::GetService<IMessageBusService>()` 查询。

## 3. 生命周期规则

上下文与模块遵循统一生命周期：

1. `LoadModules()` / `LoadModule()` 装载模块实例。
2. `Init()` 初始化资源和配置。
3. `Start()` 启动业务逻辑。
4. `Stop()` 停止业务逻辑。
5. `Fini()` 释放资源并清理运行时状态。

设计约束：

- 初始化和启动按加载顺序执行。
- 停止和释放按逆序执行。
- 批量配置加载采用“先验证、后提交”的策略，避免半加载状态。
- 服务接口注册跟随模块实例生命周期，在反初始化后统一清理。

## 4. 服务发现规则

模块实例查询与服务查询刻意分离：

- 查模块实例：`IModuleManager::Module<T>(name)`
- 查服务接口：`IContext::GetService<T>(name)` 或 `IContext::GetService<T>()`

这样做的原因：

- 模块实例表示“装载对象和生命周期单元”。
- 服务接口表示“模块对外提供的能力”。
- 一个模块可以既是模块实例，又暴露一个或多个服务接口。

唯一服务查询重载只在该接口类型恰好注册一个实例时成功；当不存在或存在多个实例时，都会返回错误，由调用方显式处理。

## 5. RabbitMQ Bus 模块

`rabbitmq_bus` 是当前仓库内置的消息总线模块，职责如下：

- 根据模块配置建立 RabbitMQ 连接。
- 启动后自动准备交换机、队列、绑定和消费者。
- 对外暴露 `IMessageBusService` 接口。
- 将消费者收到的消息分发给业务注册的处理回调。

建议使用方式：

- 用模块配置声明连接信息和拓扑结构。
- 用 `GetService<IMessageBusService>(instance_name)` 获取服务。
- 用服务接口执行发布、声明和回调注册操作。

## 6. 公开边界

以下内容属于公开契约：

- `IContext`
- `IModule`
- `IModuleManager`
- `ModuleState`
- `IMessageBusService`
- `Types.h` 中的消息模型类型

以下内容不属于公开契约：

- 具体模块实现类
- 插件工厂辅助宏
- 框架内部基类
- 服务注册表和模块内部共享状态

外部工程若直接依赖这些内部实现，后续升级时可能发生不兼容变更。
