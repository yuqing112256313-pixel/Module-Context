# Module-Context 架构评价文档

## 1. 总体评价

`Module-Context` 的架构思路是清晰的，核心定位也比较准确，它把“模块加载、生命周期、服务发现”三件事收敛在一个轻量运行时里，并通过插件动态库把具体能力隔离出去。

如果把它定义为一个 **模块化运行时内核**，我认为这个项目当前处于：

- **架构方向正确**
- **核心抽象基本成立**
- **实现可用且可验证**
- **扩展机制还有几处明显的框架级约束**

一句话评价：**它已经具备一个小型模块框架的骨架，但离“高扩展、低耦合、长期稳定演进”的成熟框架还有一段距离。**

## 2. 优点

### 2.1 公共接口边界清楚

项目把公开契约集中在 `include/module_context/**`，把内部实现放在 `src/**` 和 `modules/**`，边界是清楚的。

这有几个直接好处：

- 使用者知道该依赖哪些头文件
- 内部实现可演进，外部接口相对稳定
- 文档、示例、测试都围绕同一套公开接口展开

这是一个框架项目很重要的成熟信号。

### 2.2 角色分工合理

核心对象划分比较自然：

- `IContext` 负责统一入口
- `IModuleManager` 负责加载和编排
- `IModule` 代表模块实例
- `ServiceRegistry` 负责服务发现
- `ModuleBase` 负责生命周期模板化

这种分层能让调用方、框架层、模块实现层各自关注自己的职责，整体结构不乱。

### 2.3 生命周期编排设计是稳的

`Init/Start` 正序，`Stop/Fini` 逆序，这个选择是对的。

它符合常见模块依赖关系下的资源启停规律，也让框架在面对前后置依赖时更容易推理。

另外，`ModuleBase` 把状态机和非法状态校验封装起来，避免每个模块重复造生命周期轮子，这一点很实用。

### 2.4 配置加载有“先校验、后提交”的意识

`ModuleManager::LoadModules()` 在批量装载时采用 staged 两阶段思路，先完成配置解析、插件创建、名称注入和类型校验，再统一写入管理器状态。

这能避免批量加载过程中的半成功、半失败污染运行时状态，说明作者对框架一致性是有意识的。

### 2.5 插件 ABI 模型简单直接

通过固定导出符号：

- `GetPluginApiVersion()`
- `CreatePlugin()`
- `DestroyPlugin()`

来跨动态库边界加载插件，这种做法很朴素，但在 C++ 插件系统里是成熟且稳定的路线。

它的优点是：

- ABI 边界清楚
- 插件宿主关系容易理解
- 测试和诊断成本较低

### 2.6 测试覆盖面比很多同类小框架要好

从现有测试看，项目已经覆盖了：

- 生命周期状态流转
- 模块配置解析
- 插件装载与服务发现
- RabbitMQ 模块基础行为

这说明项目不是只停留在“接口设计层面”，而是在往可交付框架推进。

## 3. 主要问题与风险

### 3.1 服务注册机制不够开放，违背开放封闭原则

当前 `ServiceRegistry::RegisterKnownServices()` 是白名单式识别：

- 框架通过 `dynamic_cast` 判断模块是否实现某个已知服务接口
- 如果新增服务类型，需要同步修改核心框架代码

这意味着：

- 新增服务接口不是纯插件扩展，而是“插件 + 核心框架改造”
- 框架层对业务服务类型形成反向依赖
- 扩展点没有真正闭合

这是当前最明显的架构瓶颈之一。

**影响**：随着模块类型和服务类型增加，核心框架会越来越像“服务类型注册白名单中心”，维护成本会持续上升。

### 3.2 服务暴露时机与生命周期边界有点模糊

模块在 `LoadModule/LoadModules` 后就会进入 `ServiceRegistry`，因此服务在模块尚未 `Init/Start` 时就可能被查询到。

这会导致一种状态：

- 服务对象能拿到
- 但服务能力还不能安全使用
- 调用方需要自行理解 `Created / Inited / Started` 的语义

例如 `rabbitmq_bus` 在未启动时可被查到，但真正调用会返回 `kDisconnected`。

这不是错，但它让“服务已发现”和“服务可用”混在了一起。

**建议方向**：明确约束服务是在 `Load`、`Init` 还是 `Start` 后才视为可用，或者为服务增加统一健康状态/就绪状态约定。

### 3.3 生命周期失败后的回滚策略还不够完整

当前 `LoadModules()` 做了实例创建阶段的两阶段提交，但 `Init()` 或 `Start()` 一旦在中途失败，框架本身没有自动回滚到一致状态。

这意味着可能出现：

- 前几个模块已经 `Init/Start` 成功
- 后一个模块失败
- 框架把错误抛给上层
- 回收和恢复动作依赖调用方后续是否显式 `Stop/Fini`

对于框架层来说，这种行为可以接受，但不够强健。

### 3.4 `rabbitmq_bus` 模块实现过于集中

`RabbitMqBusModule.cpp` 同时承担了很多职责：

- 配置解析
- AMQP 类型转换
- socket 建连
- 心跳与重连
- 拓扑声明
- 消费者启动
- 消息投递
- ACK/NACK 完成

这让它成为一个“能工作的大文件”，但不是一个容易演进的模块。

**风险**：

- 修改一个点时容易波及其他逻辑
- 单元测试颗粒度变粗
- 后续接入别的消息中间件时难以复用设计经验

### 3.5 单例入口有便利性，但限制了更强的运行时隔离能力

`IContext::Instance()` 提供了全局默认上下文，这对示例和轻量接入很友好，但会天然带来以下问题：

- 多上下文并存能力弱化
- 全局状态更难做复杂测试隔离
- 后续如需支持多租户/多运行域，会受到 API 使用习惯牵引

虽然项目里也有 `Context` 具体类可直接实例化，这个问题还不算严重，但全局单例仍然会在长期演进中形成默认路径依赖。

### 3.6 构建阶段直接 patch 依赖源码，工程可重复性存在风险

顶层 `CMakeLists.txt` 中会对 `foundation` 源码做兼容性 patch。

这能快速解决依赖问题，但从工程治理角度看有两个风险：

- 构建结果依赖“拉取后再改写源码”的过程
- 升级依赖版本时，patch 行为可能 silently 失效或产生新兼容性问题

对于 PoC 或内部项目可以接受，但如果要长期维护，最好逐步收敛为：

- 上游修复
- 明确版本 pin
- 或可审计的 patch 文件机制

### 3.7 服务注册表保存原始指针，生命周期安全依赖框架纪律

`ServiceRegistry` 里保存的是 `IModule*` 原始指针，而真正所有权在 `ModuleManager` 的插件句柄里。

当前实现通过 `Fini()` 时统一 `Clear()` 来保证安全，逻辑上能成立，但这种安全是“约定式”的，不是“类型系统表达式”的。

如果后续出现更复杂的热更新、局部卸载、异步访问场景，这里容易成为脆弱点。

## 4. 架构成熟度判断

### 4.1 适合的场景

当前架构比较适合：

- 单进程内模块化运行时
- 中小规模插件体系
- 服务接口种类有限、团队可统一治理的项目
- 希望快速搭框架，而不是一开始就追求超高抽象能力的场景

### 4.2 暂时不太适合的场景

当前架构还不太适合：

- 高频扩展新服务类型的平台型框架
- 需要强隔离、多上下文、多租户的运行时
- 大量插件并行开发、要求框架核心长期不改的生态型项目
- 需要热插拔、热升级、细粒度卸载恢复的复杂运行时

## 5. 优先级建议

### P1，优先做

#### 5.1 让服务注册从“白名单识别”升级为“模块自描述注册”

目标是把：

- 框架知道有哪些服务类型

改成：

- 模块在加载或初始化时主动声明它提供哪些服务

可选方案：

1. 模块实现一个服务枚举接口
2. 插件工厂返回模块同时返回服务描述
3. 在 `IModule` 之外增加 `IServiceProvider`/`IServiceRegistrar`

这是最值得优先处理的框架点。

#### 5.2 拆分 `rabbitmq_bus` 的超大实现文件

建议至少拆成几块：

- `RabbitMqConfigParser`
- `RabbitMqConnectionDriver`
- `RabbitMqTopologyBootstrapper`
- `RabbitMqConsumerDispatcher`
- `AmqpValueConverter`

这样以后无论是修复问题、补测试，还是支持更多中间件，都会轻很多。

#### 5.3 明确服务可用性的正式契约

建议在框架层明确其中一种语义：

- 只要能查到就允许调用，但必须自行处理状态
- 只有 `Init` 后可查询
- 只有 `Start` 后可查询
- 查询返回服务包装对象，并带统一 readiness / health 语义

当前这块更多是“代码行为”，还不够像“框架契约”。

### P2，建议尽快规划

#### 5.4 为 `Init/Start` 失败补齐自动回滚策略

理想行为是：

- `Init()` 中途失败，自动对已初始化模块执行逆序 `Fini()`
- `Start()` 中途失败，自动对已启动模块执行逆序 `Stop()`

这样框架的一致性会更强，上层也更省心。

#### 5.5 弱化全局单例的默认地位

保留 `IContext::Instance()` 没问题，但更推荐文档和示例中同步强调显式 `Context` 实例化的方式，让未来多上下文能力不被 API 习惯锁死。

#### 5.6 处理依赖 patch 的工程化问题

建议逐步把当前 CMake 内联 patch 收敛到更可维护的方案，不然随着依赖升级，构建链会变得不透明。

## 6. 综合评分

如果按“作为一个核心框架层项目”的角度打分，我会给：

**7.8 / 10**

### 打分理由

加分项：

- 核心抽象清楚
- 生命周期模型正确
- 插件 ABI 方案稳定
- 文档与测试相对完整
- 已经形成可运行的模块化主线

扣分项：

- 服务扩展机制仍偏封闭
- 运行时失败恢复不够强
- 具体模块实现有明显大文件问题
- 构建阶段 patch 第三方依赖有长期维护风险

## 7. 最终结论

这是一个 **方向靠谱、骨架清楚、已经具备实用价值** 的模块框架。

它最值得肯定的地方，不是“功能很多”，而是它已经把：

- 模块
- 生命周期
- 服务发现
- 插件加载

这几件本来很容易混乱的事情，整理成了一套基本自洽的模型。

它最需要补强的地方，则是把“当前可用”继续推进到“长期可扩展”，尤其是：

- 服务注册机制开放化
- 生命周期失败回滚
- 复杂模块拆分治理

如果这三点补上，这个项目会从“好用的内部框架”更接近“可持续演进的框架底座”。
