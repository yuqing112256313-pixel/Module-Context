# RabbitMQ 图像链路 60 张/秒可行性汇报

## 1. 领导先看结论

### 结论 1，当前单机拟真环境已经把主要瓶颈摸清了
在当前最激进的单机拟真压测下，系统已经从最初的 **1.42 张/秒** 提升到 **26.03 张/秒**，说明：

- 旧版主要问题确实是 **主从内部实现架构不合理**
- 通过 **主从异步化、去掉输出图物化、mmap 读、并发发布、有界队列和 inflight 背压**，吞吐已经显著提升
- 但距离目标 **60 张/秒** 仍有约 **2.3 倍**差距

### 结论 2，当前最终瓶颈已经不是 worker 算法处理，也不是 RabbitMQ 队列堆积
当前最激进一轮中：

- worker `process_p50 / p95 = 23 / 50 ms`
- worker `read_p50 / p95 = 1 / 1 ms`
- task queue backlog 基本为 **0**
- task queue unacked 基本为 **0**

说明：

> **当前瓶颈不再是 worker 算法执行，也不是“队列堆爆”。**
> **主瓶颈已转移到主机侧 20MB 图像持续生成/落地，以及 RabbitMQ 控制面 publish 路径自身的高调用开销。**

### 结论 3，若真实场景是 1 主 5 从、20MB 图像、目标 60 张/秒，10G 网络不可能达标
因为仅图像数据面原始吞吐就需要：

- `20MB * 60 = 1.2 GB/s`

换算网络：

- **10GbE 理论上限约 1.25 GB/s**，实际可用通常 **1.0 到 1.1 GB/s**
- 所以 **10G 无法稳定承载 60 张/秒的 20MB 原始图像流量**

> **25G 是最低硬性网络条件，40G 更稳妥。**

### 结论 4，真实 Windows 多机要达到 60 张/秒，最低硬性条件不是“多开线程”，而是“换对数据面”
若继续沿用“主机不断生成/落 20MB 文件，再由从机读取文件”的思路，即使继续加线程，也很难把 26 张/秒直接推到 60 张/秒。

要达到 60 张/秒，最低硬性条件是：

1. **RabbitMQ 只传元数据，不传大图数据**
2. **图像数据面脱离文件热路径，改为共享内存 / 预分配大块缓冲 / 单次网络传输**
3. **主机网络至少 25G，最好 40G**
4. **1 主 5 从下，每个从机单张总服务时间必须稳定低于 83.3ms**

## 4. 当前最终瓶颈分析

## 4.1 不是 worker 算法瓶颈
当前最激进一轮：

- worker `read_p50 / p95 = 1 / 1 ms`
- worker `process_p50 / p95 = 23 / 50 ms`
- worker `cleanup_p50 / p95 = 0 / 2.1 ms`

按这个服务时间估算，单 worker 本地处理能力远高于当前实测 26 张/秒的全局完成吞吐。

> **因此，当前系统不是被 worker 算法阶段卡住。**

## 4.2 不是 RabbitMQ 队列堆积瓶颈
监控显示：

- task queue backlog 峰值约 **0**
- task queue unacked 峰值约 **0**

即使考虑 1 秒采样粒度可能漏掉短促尖峰，也说明没有形成“队列长期堆积”的形态。

> **因此，当前也不是 RabbitMQ 队列本身被堆爆。**

## 4.3 当前最重的是主机侧数据面 + publish 路径
当前最佳结果中：

- 主机写图 `P50 / P95 = 21 / 27 ms`
- 主机 publish 调用 `P50 / P95 = 254.5 / 294.2 ms`
- 队列等待 `P50 / P95 = 256 / 297.1 ms`

这里有一个非常关键的现象：

> `queue_wait` 和 `publish_call_ms` 几乎同量级。

这说明当前大量“等待”并不是消息已经进入队列后排很久，而更像是：

- Publish 调用本身就重
- 消息进入 broker / 消费者回调这条控制路径仍然较慢
- 或者 Publish 内部包含较重的同步推进、锁、flush、事件循环推进成本

### 结论
当前单机拟真环境的最终瓶颈排序可归纳为：

1. **主机侧 20MB 图像持续生成 / 落地 / 页面缓存竞争**
2. **RabbitMqBus publish 路径自身调用开销偏高**
3. **单机把 master、5 workers、RabbitMQ、文件映射、页缓存竞争全部叠到一台机器上**

## 5. 要达到目标速率 60 张/秒，需要提升哪些

## 5.1 当前单机拟真路径，需要提升的量化差距
基于当前最佳结果：

- 当前完成吞吐：**26.03/s**
- 目标完成吞吐：**60/s**
- 需要提升倍数：`60 / 26.03 ≈ 2.30`

也就是说：

> **当前单机文件/文件映射拟真数据面，需要整体再提升约 2.3 倍，才有机会触到 60/s。**

但从当前瓶颈位置看，继续单纯加线程，收益会快速递减。

## 5.2 要提升到 60/s，最关键的改造项

### 改造项 A，数据面从“文件热路径”切到“共享内存/预分配缓冲热路径”
当前 master 仍然在持续生成 20MB 图像文件，这是非常重的路径。

建议：
- 主机预分配固定 slot ring buffer
- 图像直接写入共享内存 / 大页缓冲
- worker 通过句柄 / 元数据定位数据
- 不再把 20MB 图像作为文件热路径的一部分

### 改造项 B，RabbitMQ 控制面进一步轻量化
当前 `publish_call_p50 ≈ 254.5ms` 仍然太重。

若仍使用 8 条并行发布 lane，理论上想达到 60/s，需要：

- 平均每 lane publish 周期 <= `8 / 60 = 133ms`

而当前是：
- `publish_call_p50 ≈ 254.5ms`

这意味着仅从 publish 路径角度看，也还需要再优化约 **1.9 倍**。

### 改造项 C，1 主 5 从下，每台从机总服务时间必须低于 83.3ms
这是硬数学条件：

- 全局目标 60/s
- 5 台从机平均每台需要处理 12/s
- 则每台从机单张平均总服务时间必须 <= `1 / 12 = 83.3ms`

因此：

> **如果真实算法 + 数据访问 + 结果发布的单张总服务时间超过 83ms，1 主 5 从就不可能稳定达到 60/s。**

### 改造项 D，必须保留背压，但背压不能保守到压死资源
本轮结果说明：

- 无界异步会导致排队和延迟失控
- 有界队列 + inflight 背压能显著提高有效吞吐

因此真实系统中建议：
- 有界 ring buffer
- 有界 inflight credit
- 回收后再补任务
- 让系统始终在高利用率区间运行，但不让队列无限积压

## 7. Windows 多机真实场景预测报表

### 7.1 预测前提
目标场景：
- **1 主 5 从**
- **Windows 多机**
- CPU：**i9-12900K**
- 内存：**192GB**
- 网络：**10G ~ 40G**
- RabbitMQ 只传轻量任务/结果元数据
- 图像大小：**20MB**
- 目标吞吐：**60 张/秒**

### 7.2 先看最硬的网络条件
原始图像数据面吞吐：

- `20MB * 60 = 1.2 GB/s`

网络结论：

| 网络 | 理论情况 | 能否稳定支撑 60/s |
|---|---|---|
| 10G | 理论极限约 1.25GB/s，实际可用通常 1.0~1.1GB/s | **不能** |
| 25G | 理论约 3.125GB/s，实际可用通常 >2.5GB/s | **可以，且是最低硬性条件** |
| 40G | 理论约 5GB/s | **可以，且更有安全余量** |

> **若 20MB 原始图像要跨主从网络传输，10G 是硬性不够的。25G 是最低网络门槛。**

### 7.3 再看 1 主 5 从的容量硬条件
1 主 5 从，目标 60/s，则每台从机平均 12/s。

所以单张总服务时间硬条件：

- `1 / 12 = 83.3ms`

建议工程口径不要只看平均值，而看稳定值：

- 平均总服务时间最好 <= **70ms**
- P95 总服务时间最好 <= **80ms**
- 否则 60/s 只能偶发冲到，不能稳态承诺

### 7.4 Windows 多机场景下的预测吞吐
以下预测分两类前提：

#### 前提 A，仍然沿用“文件/共享目录风格”的重数据路径
即使搬到多机，性能会比单机好，但仍会吃到：
- 文件元数据
- 文件落盘 / 缓存
- 网络文件系统或显式文件同步成本

#### 前提 B，升级为“共享内存 / 预分配缓冲 / RabbitMQ 只传元数据”的目标数据面
这是更接近最终落地的合理方案。

### 7.5 预测表

| 真实部署方案 | 网络 | 预测稳定完成吞吐 | 对 60/s 的判断 |
|---|---|---:|---|
| 1 主 5 从，沿用重文件路径 | 10G | 20 ~ 35/s | 明显不够 |
| 1 主 5 从，沿用重文件路径 | 25G | 25 ~ 40/s | 仍明显不够 |
| 1 主 5 从，沿用重文件路径 | 40G | 30 ~ 45/s | 仍不建议承诺 |
| 1 主 5 从，优化控制面但未换数据面 | 25G | 35 ~ 50/s | 接近但不稳 |

| **1 主 5 从，目标数据面（共享内存/预分配缓冲）** | **25G** | **55 ~ 65/s** | **有机会达标，是最低可行档** |
| **1 主 5 从，目标数据面（共享内存/预分配缓冲）** | **40G** | **60/s 可稳态承诺概率更高** | **推荐档** |

## 8. 达到理论速率 60/s 的最低硬性条件

这是最重要的一页，可以直接给领导。

### 硬性条件 1，网络必须至少 25G
原因：
- 20MB * 60/s = 1.2GB/s 原始数据面吞吐
- 10G 实际可用吞吐不够

### 硬性条件 2，RabbitMQ 只能传元数据，不能让大图走消息总线
RabbitMQ 适合：
- `task_id`
- `buffer_id / slot_id`
- `offset`
- `size`
- `generation`
- `checksum`

不适合：
- 20MB 图像内容本体

### 硬性条件 3，数据面必须脱离文件热路径
必须避免：
- 每张图都创建文件
- 每张图都落盘
- 从机再读文件
- 再写输出图

最低可行实现应为：
- 预分配大块缓冲区 / 共享内存 / pagefile-backed mapping / section object
- worker 直接定位并读取缓冲区数据
- 结果只回轻量元数据

### 硬性条件 4，1 主 5 从下，每个从机单张总服务时间必须 <= 83.3ms
这是硬上限。

若要稳态承诺 60/s，更建议：
- 平均 <= **70ms**
- P95 <= **80ms**

### 硬性条件 5，控制面 publish 路径必须明显轻于当前 harness
当前单机最优里：
- `publish_call_p50 ≈ 254.5ms`

这对 60/s 来说仍然太重。

最低要求应为：
- publish 路径进一步异步化
- 单次控制消息 publish 周期显著下降
- 或减少 Publish 内部同步等待
- 目标是把当前 publish 路径再优化至少 **1.5 ~ 2 倍**

---

## 9. 建议的最终落地方向

### 推荐方案
- RabbitMQ：只做轻量控制面
- 数据面：预分配共享内存 / 大块缓冲 / 零拷贝风格传输
- master：生成 -> 写入 buffer -> 发布元数据 -> 回收 credit
- worker：消费元数据 -> 直接读 buffer -> 处理 -> 发布轻量结果 -> 回收 slot
- 全链路：有界队列 + inflight credit 背压

### 不建议继续投入的方向
- 继续在“文件落盘 / 文件映射拟真”路径上无上限加线程
- 继续把 20MB 图像当成普通文件流转主体

因为从当前数据看，这条路径已经接近收益递减区。
