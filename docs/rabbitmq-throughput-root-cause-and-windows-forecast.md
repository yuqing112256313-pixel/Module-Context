# RabbitMQ 图像链路瓶颈归因与 Windows 实机预测报表

## 一页结论

### 结论 1, 旧版瓶颈不主要是 RabbitMQ，也不单纯是“本机磁盘慢”
旧版单机压测之所以严重堵塞，核心原因是 **数据路径总吞吐过大 + 实现里存在重复大拷贝和重复落盘**。

在 20MB 图像、60 张/秒目标下，旧链路实际接近三段大流量叠加：

- 主机写输入图: `60 * 20MB = 1.2 GB/s`
- worker 读输入图: `1.2 GB/s`
- worker 再写 20MB 输出图: `1.2 GB/s`

合计接近 **3.6 GB/s 级别** 的单机文件 IO 压力，还没有算：

- 文件创建 / 删除元数据成本
- 页缓存竞争
- 用户态整图 memcpy
- RabbitMQ 同步 publish / ack
- Colima + Docker 虚拟化开销

### 结论 2, 主要收益来自“去掉内部实现瓶颈”，不是 mmap 单点魔法
对比结果很清楚：

- 旧版单机 real perf: 完成吞吐约 **1.42 张/秒**
- 优化后 stream 基线: 完成吞吐约 **7.78 张/秒**
- 优化后 mmap: 完成吞吐约 **7.21 张/秒**

也就是说，**先把重复大拷贝、重复落盘、主从内部不必要的串行化去掉**，收益远大于单纯把文件 API 从 stream 改成 mmap。

### 结论 3, mmap 明显改善了读路径和尾延迟，但当前上限已经转移到控制面和流水线组织
- stream `read_p50 / p95`: **95 / 96 ms**
- mmap `read_p50 / p95`: **1 / 1 ms**

但总吞吐没有同比例继续上升，说明新瓶颈已经转移到：

- master 同步 publish 节奏
- 控制面串行化
- worker 固定模拟处理耗时
- 整体流水线组织

### 结论 4, 1 主 5 从, 80ms 处理耗时的配置，理论上就非常难稳定达到 60 张/秒
若 5 个 worker 每张图的总服务时间为 `T`，则系统理论吞吐上限约为：

`吞吐上限 = 5 / T`

要稳定达到 60 张/秒，需要：

`T <= 5 / 60 = 83.3 ms`

而当前优化后的 mmap 结果里：

- `process_p50 = 119.5 ms`
- `process_p95 = 155 ms`

因此，**在当前 5 worker + 80ms 模拟处理的设定下，60 张/秒几乎没有足够 headroom**。

## 当前测试环境条件

### 环境拓扑
- 单机 macOS 本地测试
- 测试主机: **Apple M4 Max**, **36GB RAM**, **14 CPU 核心**
- RabbitMQ 跑在 Colima + Docker 中
- 1 个 master 进程 + 5 个 worker 进程，全都在同一台机器上
- RabbitMQ 仅承担轻量控制消息
- 图像通过本地文件 / 文件映射方式做拟真
- 当前对比测试已关闭“写 20MB 输出图”这条重 IO 路径，更接近真实数据面设计

### 当前测试参数
- 图像大小: **20,971,520 bytes**, 约 20MB
- 目标速率: **60 张/秒**
- worker 数量: **5**
- 模拟处理耗时: **80ms / 张**
- 结果回传: 轻量文本消息

### 当前环境为什么天然偏保守
单机环境会引入真实部署中不会叠加得这么重的干扰：

1. master / worker / RabbitMQ 共享同一台机器的 CPU、内存带宽、页缓存
2. 输入写入和 worker 读取争用同一块本地文件系统
3. Colima + Docker 带来额外虚拟化开销
4. 当前 mmap 仍然是**文件后备映射**，并不等于 Windows 的**页文件后备共享内存**

因此，这套结果对真实 6 台 Windows 实机来说偏保守，但它仍然非常有价值，因为它暴露了真正的控制面和架构瓶颈。

## 实测结果汇总

### A. 旧版单机 real perf, 作为问题基线
- 场景: `build/tests/rabbitmq_perf_runtime`
- 完成吞吐: **约 1.42 张/秒**
- 发布吞吐: **约 3.63 张/秒**
- 端到端延迟 P95: **约 50s**
- 根因: 重复文件写读 + 额外输出图写入 + 单机页缓存竞争 + 同步控制面

### B. 优化后 stream 基线
- 场景: `build/tests/rabbitmq_perf_runtime_stream_cmp`
- 成功率: **60 / 60**
- 发布吞吐: **7.87 张/秒**
- 完成吞吐: **7.78 张/秒**
- `image_write_p50 / p95`: **22 / 25.05 ms**
- `publish_call_p50 / p95`: **65 / 273.1 ms**
- `read_p50 / p95`: **95 / 96 ms**
- `queue_wait_p50 / p95`: **207 / 411.25 ms**
- `process_p50 / p95`: **155 / 198.25 ms**
- `end_to_end_p50 / p95 / p99`: **531 / 824.25 / 980.51 ms**

### C. 优化后 mmap
- 场景: `build/tests/rabbitmq_perf_runtime_mmap_cmp`
- 成功率: **60 / 60**
- 发布窗口: **8.56s**
- 完成窗口: **8.317s**
- 发布吞吐: **7.01 张/秒**
- 完成吞吐: **7.21 张/秒**
- `image_write_p50 / p95`: **19 / 20.05 ms**
- `publish_call_p50 / p95`: **123.5 / 258.35 ms**
- `read_p50 / p95`: **1 / 1 ms**
- `queue_wait_p50 / p95`: **167 / 263.15 ms**
- `process_p50 / p95`: **119.5 / 155 ms**
- `end_to_end_p50 / p95 / p99`: **455.5 / 577.2 / 611.56 ms**

### 对比结论
- mmap 对 **读路径和尾延迟** 有显著帮助
- 但吞吐没有同比例继续上升，说明系统上限已经转移到 **主线程 publish 节奏 + 控制面串行化 + worker 服务时间**

## 当前真实瓶颈归因

### 瓶颈 1, 旧版数据面总流量设计过重
旧版的真正问题不是“20MB 文件本身太大”，而是同一张图在链路内被多次“落盘 / 读取 / 再写回”。

在最终架构里，图像这条路径本来就不应该经过：
- 整图落盘
- 整图二次读取
- 再物化整图输出

而应该是：
- 控制信息走 RabbitMQ
- 大图数据走共享内存 / 映射对象
- 结果只回轻量元数据

### 瓶颈 2, master 侧当前仍是串行 publish
现在 master 还是“一边写图，一边同步 publish”，这会直接限制发任务速率。

从结果看：
- stream `publish_call_p50 = 65ms`
- mmap `publish_call_p50 = 123.5ms`

如果控制面仍按这种串行方式做，单线程理论发布上限远低于 60 张/秒。

### 瓶颈 3, 5 个 worker + 80ms 处理耗时，本来就接近极限
这是最关键的数学事实。

假设算法处理本身就是 80ms，那么 5 个 worker 的绝对理论上限只有：

`5 / 0.08 = 62.5 张/秒`

这意味着要达到 60 张/秒，所有额外开销总和只能占：

`83.3ms - 80ms = 3.3ms`

这在真实系统里几乎不可能。

所以，**如果配置固定为 1 主 5 从，并且单张处理耗时确实是 80ms 级别，那么 60 张/秒从容量设计上就过于激进。**

### 瓶颈 4, 单机测试确实放大了文件系统和页缓存瓶颈
答案是 **是的**，单机测试会额外放大：
- 本地文件系统吞吐压力
- 页缓存命中 / 驱逐竞争
- master 写 + worker 读同盘干扰
- 容器 RabbitMQ 与业务进程争资源

这会让单机结论更悲观。

## 达到 60 张/秒需要什么条件

### 条件 1, 数据面必须从“文件 IO”升级成“共享内存数据面”
建议最终方案：
- RabbitMQ 只发轻量消息: `task_id / mapping_name / offset / size / generation / checksum`
- 图像数据走 Windows 共享内存 / section object
- worker 直接 map view, 原地读取处理
- 结果只回轻量元数据, 不回整图

### 条件 2, 控制面必须异步化
当前 master 侧同步 publish 不能代表最终设计。

要打到 60 张/秒，控制面至少要做到：
- 多 buffer / 多 slot 预分配
- 图像写入与 publish 解耦
- 异步 publish
- 回收线程单独处理 lease / refcount

### 条件 3, 5 个 worker 不足以“稳态保障”60 张/秒
按 worker 总服务时间推算：

- 若单张服务时间 **90ms**, 需要 worker 数 `60 * 0.09 = 5.4`, 至少 **6 个 worker**
- 若单张服务时间 **100ms**, 至少 **6 个 worker**
- 若单张服务时间 **120ms**, 至少 **8 个 worker**
- 若单张服务时间 **155ms**, 至少 **10 个 worker**

因此，若要给领导一个稳妥结论：

> **在 80ms 算法耗时设定下，1 主 5 从想长期稳定达到 60 张/秒，容量余量明显不足。**
> **建议至少按 1 主 7 到 9 从设计，或者把单张总服务时间压到 83ms 以下。**

### 条件 4, 主机网络链路至少需要 25G
20MB * 60 张/秒 = **1.2 GB/s** 主机总出流量。

换算为网络：
- 10GbE 理论约 1.25 GB/s，实际可用通常 **1.0 到 1.1 GB/s**，**达不到 1.2 GB/s 的安全线**
- 25GbE 理论约 3.125 GB/s，实际可用一般 > 2.5 GB/s, **足够**
- 40GbE 更充裕

所以：

> **若 20MB 图像真的要稳定做到 60 张/秒，主机到 worker 集群的数据面总出口建议至少 25G。**

## 6 台 Windows 实机, 10 到 40G 带宽下的预测

预测假设：
- 1 台 master + 5 台 worker
- 图像数据面切换为 Windows 共享内存 / 映射对象风格
- RabbitMQ 仍只传控制消息
- 不再物化 20MB 输出图
- 算法处理耗时保持 **80ms** 级别
- 控制面做基本异步化，但不假设激进优化到零开销

### 先看网络理论上限
按主机总出口 1.2 GB/s 计算：

- 10G: 原始理论上限约 **50 到 55 张/秒**
- 25G: 网络不是瓶颈
- 40G: 网络不是瓶颈

### 再看 worker 容量上限
按更接近真实部署的共享内存实现估算：
- 单 worker 总服务时间大概率会落在 **90 到 120ms** 区间
- 5 个 worker 的总容量大约在 **41 到 55 张/秒**

### 预测表

| 场景 | 主要瓶颈 | 预测持续吞吐 | 结论 |
|---|---|---:|---|
| 6 台 Windows, 10G, 1 主 5 从 | 主机总出口 + worker 容量共同限制 | **40 到 50 张/秒** | 大概率达不到 60 张/秒 |
| 6 台 Windows, 25G, 1 主 5 从 | 主要是 worker 容量 | **45 到 55 张/秒** | 逼近但仍不稳 |
| 6 台 Windows, 40G, 1 主 5 从 | 主要是 worker 容量 | **45 到 58 张/秒** | 最好情况下接近，但不建议承诺稳态 60 |
| 6 台 Windows, 25G/40G, 1 主 7 到 9 从 | 控制面与算法耗时 | **60 张/秒可达** | 更合理的容量设计 |

### 领导可直接看的判断

> **如果按 6 台 Windows 实机、1 主 5 从部署，即使网络提升到 25G 到 40G，60 张/秒仍然不应被当成稳态承诺值。**
> **更现实的稳态预测区间是 45 到 55 张/秒。**
> **若要把 60 张/秒作为工程目标，建议同时满足两件事：**
> 1. **主机数据面至少 25G**
> 2. **worker 数扩到 7 到 9 台，或者把单张总服务时间压到 83ms 以下**

## 建议的最终架构

### 数据面
- 用 Windows 共享内存 / section object 传大图
- 不再让整图经过 RabbitMQ
- 不再默认物化整图输出

### 控制面
- RabbitMQ 仅发小消息
- task / result 都只带元数据
- publish 必须异步化

### 恢复能力
- 单独保留一个小型持久化元数据日志
- 记录 task 生命周期、mapping 元信息、generation、checksum、lease/refcount
- 快路径走共享内存，恢复路径走小日志

---

## 对领导的最终建议

1. **当前单机结果不能直接代表 6 台 Windows 实机上限，单机测试放大了文件系统和页缓存瓶颈。**
2. **真正暴露出来的问题不是 RabbitMQ 本身，而是旧数据路径有重复大拷贝和重复落盘。**
3. **在去掉这些内部瓶颈后，性能已经从 1.42 张/秒提升到 7.x 张/秒，说明架构方向是正确的。**
4. **mmap 进一步把读路径延迟显著压低，但要冲 60 张/秒，下一阶段重点不再是文件 API，而是共享内存数据面、异步控制面和 worker 容量设计。**
5. **若目标是稳态 60 张/秒，不建议按 1 主 5 从承诺，建议按 1 主 7 到 9 从、25G 以上数据面链路来设计。**
